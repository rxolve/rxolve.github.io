> 정리하며 책의 문장을 요약해서 옮깁니다.

- 코드 정리는 어디서 해야 할까요?
- 볼썽사나운 모습이 이어지는 한 가지 사례입니다.
  - 동작 변경 코드와 함께 제가 만든 코드 정리 내용을 넣습니다.
  - 검토하던 사람들이 제가 만드 PR이 너무 길다고 불평합니다.
  - 제가 만든 코드 정리 내용을 분리해서, 동작 변경 PR 앞이나 뒤에 둡니다.
  - 검토하는 사람들이 코드 정리만 담긴 PR에 대해 무슨 의미로 만들었는지 모르겠다고 불평합니다.
  - 다시 1로 돌아갑니다.
- 코드 정리를 하면 어딘가에서는 처리해야 합니다. 그러면 어디서 해야할까요?
- 코드 정리는 별도의 PR로 만들고, 가급적 PR당 몇 개의 코드 정리만 넣습니다.
- 심지어 하나의 정리만 있더라도 PR 한 개로 만듭니다.
- 동작 변경 역시 별도의 PR로 만듭니다.
- 코드 정리와 동작 변경 사이를 번갈아 가면서 전환할 때마다 새 PR을 열어야 합니다.
  > B / SS / B / SSS / BB / S
  - B: 동작 변경 PR
  - S: 코드 정리 PR
- 작은 PR은 보통 검토 시간 단축으로 환영 받습니다.
  - 코드를 신속히 검토할 수 있으므로, 더 작은 PR을 만들도록 동기부여가 되기도 합니다.
  - 초점이 분명할수록 PR은 더 빠른 검토를 장려합니다.
- 아래의 상황에 용기를 내서 코드 정리 PR을 따로 검토하지 않는 시도를 해보세요.
  - 코드 정리에 익숙해지고
  - 작은 단위로 작업하고
  - 절대적으로 안전하게 작업하는데 익숙해지면
- 그러면 대기 시간이 추가로 줄어들어, 더욱더 작게 PR을 만들도록 장려하는 효과가 생길 겁니다.

> 오늘의 교훈: PR은 작게 쪼개서 올리기!
