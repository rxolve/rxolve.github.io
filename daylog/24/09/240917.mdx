> 조엘 온 소프트웨어 4장을 2부로 나누어 정리합니다.

21세기를 살아가는 전문 개발자인 여러분이 문자, 문자집합, 인코딩, 유니코드를 모른다면 당신을 체포한 다음에 잠수함에서 6개월 동안 양파 껍질을 까는 벌칙을 줄 겁니다. 맹세합니다.

이런 개념은 결코 어렵지 않습니다.

## 역사적인 관점

유닉스를 처음 고안하고 K&R이 The C Programming Language를 집필한 아름답던 시절에는 특수 기호가 붙을 필요가 없는 영문자만 처리해도 충분했습니다.

모든 글자를 32에서 127사이의 숫자를 사용해서 표현할 수 있는 ASCII 라고 불리는 코드만 있었습니다.

공백은 32, 글자 A는 65, 소문자 a는 97로 나타냅니다. 편리하게 7비트로 글자를 저장할 수 있었습니다.

요즘 대다수 컴퓨터가 8비트인 바이트 단위를 사용하므로, 너무나도 많은 사람이 동시에 128에서 255에 이르는 공간에 무엇을 넣어야할지 동상이몽을 한다는 문제점이 생겼습니다.

마침내 이런 난투극은 ANSI 표준 위원회에서 ANSI 표준을 체계적으로 정리함에 따라 모든 사람이 128미만 문자에 대해 어떻게 다를지에 동의하면서 끝났습니다.

그러나 여전히 지역에 따라 128이상 문자를 다루는 여러가지 방법이 존재했습니다.

문자열을 다른 컴퓨터로 옮기거나 컴퓨터 하나에서 언어를 둘 이상 처리하지 않는 경우에는 문제가 없었습니다.

하지만 인터넷 세상이 열리면서 다른 문자열로 이동하는 일이 너무나도 자연스러워지면서, 모든 가정에 금이 가기 시작했습니다.

## 유니코드

유니코드는 지구상에서 존재하는 모든 이성적인 쓰기 시스템은 물론이고 스타트렉에 나오는 클링온과 같이 공상에서나 나올 법한 언어까지 모두 포함하는 단일 문자 집합을 만들기 위한 용감한 노력 끝에 나온 결과물입니다.

유니코드가 각 글자마나 16비트를 차지하기에 65,536개만 표현할 수 있는 단순한 16비트 문자 집합이라고 오해할 수 있습니다.

실제 이런 생각은 착각이며 가장 흔한 미신이므로, 이렇게 생각했더라도 부끄러워할 필요는 없습니다.

실제로 유니코드는 글자를 다루는 일종의 철학이므로, 사물을 생각하는 유니코드 방식을 이해할 필요가 있습니다.

유니코드에서 글자는 코드 포인트라는 단순히 이론적인 개념으로 사상합니다.

유니코드에서 글자 A는 관념적인 이상이며, 공중에 둥둥 떠다닙니다.

이 관념적인 A는 B와 다르며, a와도 다릅니다. 하지만 **A**, _A_ 와는 동일합니다.

유니코드 컨소시엄은 각 알파벳에 존재하는 관념적인 철자마다 고유 번호를 붙여놓았고, U+0645라는 식으로 표현합니다.

이 고유 번호를 코드 포인트라고 부릅니다. U+는 유니코드를 의미하며, 숫자는 16진수로 표현합니다.

[유니코드 웹사이트](https://home.unicode.org)를 방문해서 각 글자마다 할당된 유니코드 코드 포인트를 확인할 수 있습니다.

실질적으로 유니코드가 정의할 수 있는 글자 개수에는 제한이 없습니다.

사실 이미 코드 숫자가 65,536개를 넘었기 때문에, 모든 유니코드 글자는 실제로 2바이트로 압축할 수 조차 없습니다.

아무튼 16비트 유니코드는 미신입니다.

Hello를 유니코드로 표현할 경우 다음과 같은 코드 포인트 5개로 대응시킬 수 있습니다.

```plaintext
U+0048 U+0065 U+006C U+006C U+006F
```

단순히 코드 포인트 묶음입니다. 정말 숫자뿐이군요.

아직 이런 문자열을 어떻게 이메일 메시지에서 표현하거나 메모리에 올리는지 이야기하지 않았습니다.

> 다음 포스팅에서는 인코딩에 대해 정리하겠습니다.

## ASCII vs ANSI vs Unicode

#### ASCII (American Standard Code for Information Interchange)

- **출시 연도**: 1963년
- **비트 수**: 7비트 인코딩
- **문자 수**: 최대 128개 (2^7 = 128)
- **주요 특징**:
  - 영어 알파벳(대소문자), 숫자, 기본 구두점, 제어 문자 등을 표현합니다.
  - 전 세계적으로 사용되며, 문자 코드의 기본으로 여겨집니다.
  - 대부분의 초기 컴퓨터 시스템에서 표준으로 채택되었습니다.
  - **한계점**: 7비트로 표현되므로 영어 외의 다른 언어를 포함한 다국어 지원에 한계가 있습니다.

#### ANSI (American National Standards Institute)

- **출시 연도**: 1980년대 (미국 표준 협회에서 정의)
- **비트 수**: 일반적으로 8비트 인코딩 (최대 256개 문자, 2^8 = 256)
- **문자 수**: ASCII를 확장하여 더 많은 문자를 포함합니다.
- **주요 특징**:
  - 128개 이상의 추가 문자들이 포함되어 있어, 영어 외에 유럽 언어(예: 프랑스어, 독일어 등)의 특수 문자, 악센트가 있는 글자를 표현할 수 있습니다.
  - 각 운영체제나 지역에 따라 ANSI의 해석과 지원하는 문자 세트가 다릅니다.
  - **한계점**: 지역에 따라 차이가 발생하므로, 일관된 문자 인코딩이 어려운 경우가 있었습니다. 이는 나중에 유니코드로 발전하는 과정에서 해결되었습니다.

#### Unicode

- **출시 연도**: 1991년
- **비트 수**: 16비트 이상 지원
- **문자 수**: 143,000개 이상의 문자 (현재는 16비트에서 확장되어 더 많은 문자를 포함)
- **주요 특징**:
  - 전 세계 모든 언어의 문자를 표현할 수 있습니다.
  - 한글, 중국어, 일본어와 같은 복잡한 문자를 비롯해 이모지(emoji)까지 포함한 매우 광범위한 문자 세트를 지원합니다.
  - **코드포인트(Code Point)**: 유니코드는 각 문자를 고유한 코드포인트로 정의하며, 예를 들어 'A'는 U+0041로 표현됩니다.
  - **인코딩 방식**: UTF-8, UTF-16, UTF-32와 같은 여러 인코딩 방식을 사용해 저장 및 전송됩니다. UTF-8은 가장 널리 쓰이며, 가변 길이로 효율적인 저장이 가능합니다.

### 주요 차이점 요약

| 항목            | **ASCII**      | **ANSI**              | **유니코드(Unicode)**                               |
| --------------- | -------------- | --------------------- | --------------------------------------------------- |
| **출시 연도**   | 1963년         | 1980년대              | 1991년                                              |
| **비트 수**     | 7비트          | 8비트                 | 16비트 이상                                         |
| **문자 수**     | 128개          | 256개                 | 143,000개 이상                                      |
| **지원 언어**   | 영어           | 영어 + 유럽 언어      | 전 세계 모든 언어 + 이모지                          |
| **인코딩 방식** | 고정 길이      | 고정 길이             | UTF-8, UTF-16, UTF-32 (가변 및 고정 길이 모두 지원) |
| **확장성**      | 매우 제한적    | 일부 유럽어 확장 가능 | 무한한 확장 가능                                    |
| **대표성**      | 영어 기반 문서 | 지역마다 다름         | 모든 언어, 문자, 심볼, 특수 기호                    |
