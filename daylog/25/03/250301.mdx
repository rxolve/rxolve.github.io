쿠버네티스는 컨테이너화된 애플리케이션을 대규모로 배포, 운영 및 관리하기 위한 오케스트레이션 플랫폼입니다.

전체 아키텍처는 크게 Control Plane(마스터 노드)와 Worker Node(워커 노드)로 나뉘며, 각 영역은 서로 긴밀하게 협력하여 클러스터의 상태를 관리하고 원하는 상태를 유지합니다.

## Control Plane (마스터 노드)

Control Plane은 클러스터 전체의 상태를 관리하고, 작업을 오케스트레이션하는 중심 역할을 합니다. 주요 구성 요소는 다음과 같습니다.

- kube-apiserver  
  모든 클러스터 내 컴포넌트와 사용자 요청을 받는 진입점입니다. RESTful API를 통해 클러스터의 상태를 조회하거나 변경할 때 사용됩니다.

- etcd  
  클러스터의 모든 설정과 상태 정보를 저장하는 분산형 키-값 저장소입니다. 클러스터 구성 및 상태 변경 사항을 안정적으로 관리합니다.

- kube-scheduler  
  새로운 파드(Pod)를 적절한 워커 노드에 배치하는 역할을 합니다. 각 노드의 리소스 상황, 정책, taint/toleration, 노드 affinity 등을 고려하여 스케줄링 결정을 내립니다.

- kube-controller-manager  
  클러스터 내 다양한 컨트롤러(노드 컨트롤러, 레플리케이션 컨트롤러 등)를 실행하여 클러스터의 실제 상태를 원하는 상태와 일치시키도록 관리합니다.

## Worker Node (워커 노드)

워커 노드는 실제 애플리케이션 컨테이너를 실행하는 호스트로, Control Plane의 지시에 따라 작업을 수행합니다. 주요 구성 요소는 다음과 같습니다.

- kubelet  
  각 워커 노드에서 실행되며, kube-apiserver와 지속적으로 통신하여 해당 노드의 파드 상태를 보고하고 관리합니다.

- kube-proxy  
  클러스터 내 네트워킹을 담당하여, 노드 내 혹은 노드 간의 파드들이 서로 통신할 수 있도록 네트워크 프록시 및 로드밸런싱 역할을 수행합니다.

- Container Runtime  
  컨테이너(예: Docker, containerd 등)를 실제로 실행시키는 엔진입니다. 쿠버네티스는 다양한 컨테이너 런타임을 지원하여 일관된 환경에서 애플리케이션이 실행될 수 있도록 합니다.

## 쿠버네티스 아키텍처의 특징 및 장점

- 확장성 & 자가 치유  
  스케줄러와 컨트롤러 매니저가 지속적으로 클러스터 상태를 모니터링하며, 이상 상황 발생 시 자동으로 복구하거나 새로운 인스턴스를 배치합니다.

- 선언적 구성  
  YAML 파일 등 선언적 구성 파일을 통해 클러스터 리소스의 원하는 상태를 정의하면, 컨트롤 플레인이 이를 자동으로 이행합니다.

- 유연한 확장과 관리  
  워커 노드와 컨트롤 플레인의 분리 덕분에, 클러스터의 일부를 독립적으로 확장하거나 업그레이드할 수 있습니다.
