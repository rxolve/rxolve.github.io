리버스 프록시(Reverse Proxy)는 클라이언트와 실제 백엔드 서버 사이에서 중계자 역할을 수행하여 트래픽을 제어·최적화하는 서버입니다.  
이를 통해 로드 밸런싱, 캐싱, SSL 종료(SSL Termination), 보안 필터링 등의 기능을 중앙화하고, 백엔드 서버의 IP를 감추어 보안성을 높이며, 성능과 가용성을 개선할 수 있습니다.  
리버스 프록시는 단일 진입점으로 기능함으로써 운영 관리가 쉬워지고 모니터링 지점이 집중되어 장애 대응이 용이해집니다.  
그러나 단일 장애점(SPOF)이 될 수 있고 구성·운영 복잡도가 증가하며, 잘못된 설정 시 오히려 병목이 될 수 있으므로 도입 시 주의가 필요합니다.

## 개념

#### 리버스 프록시 정의

리버스 프록시는 클라이언트가 보내는 요청을 받아 적절한 백엔드 서버에 전달하고, 응답을 다시 클라이언트로 반환하는 중계 서버입니다.  
클라이언트 입장에서는 리버스 프록시가 마치 실제 웹 서버인 것처럼 보입니다.

#### 일반 프록시와의 차이

일반(포워드) 프록시는 클라이언트를 보호하기 위해 클라이언트 측에 위치하고, 리버스 프록시는 서버를 보호하기 위해 서버 측에 위치합니다.

## 작동 원리

#### 요청 흐름

1. 클라이언트가 리버스 프록시에 HTTP/HTTPS 요청을 보냅니다.
2. 리버스 프록시는 로드 밸런싱 정책이나 URL 매핑 규칙에 따라 최적의 백엔드 서버를 선택합니다.
3. 선택된 백엔드 서버로 요청을 전달하고, 받은 응답을 다시 클라이언트로 반환합니다.

#### 주요 기능

- 로드 밸런싱: 여러 대의 서버에 부하를 분산하여 과부하를 방지하고 가용성을 높입니다.
- 캐싱: 정적 콘텐츠나 자주 조회되는 데이터를 캐시하여 백엔드 서버 요청을 줄이고 응답 속도를 개선합니다.
- SSL 종료: SSL/TLS 복호화를 프록시에서 처리해 백엔드 서버의 부하를 줄이고 인증서 관리를 중앙화할 수 있습니다.
- 보안 필터링: 웹 애플리케이션 방화벽(WAF) 연동, IP 필터링, 인증·인가 등을 통해 직접적인 공격을 차단합니다.
- 로그·모니터링: 모든 트래픽이 프록시를 통과하므로 중앙 집중형 로그 수집이 가능해집니다.

## 장점

- 보안 강화: 백엔드 서버 IP를 숨기고 공격 표면을 축소합니다.
- 성능 최적화: 캐싱 및 콘텐츠 압축으로 응답 속도를 높이고 네트워크 부하를 경감합니다.
- 운영 효율성: 인증·인가, SSL 관리, 로깅 등 기능을 중앙화해 개별 서버 설정 부담을 줄입니다.
- 확장성: 트래픽 증가 시 백엔드 서버를 수평 확장만으로도 대응이 용이합니다.

## 단점 및 고려사항

- 단일 장애점: 프록시 서버 장애 시 전체 서비스가 중단될 수 있으므로 이중화 구성이 필수입니다.
- 구성 복잡도: 다양한 기능 설정과 정책 관리를 위해 학습 곡선이 존재합니다.
- 레이턴시 증가: 중계 과정이 추가되어 잘못 구성된 경우 오히려 응답 시간이 늘어날 수 있습니다.

## 구현 예시

- Nginx: `proxy_pass` 지시어를 이용해 간편하게 리버스 프록시를 구성할 수 있으며, 다양한 모듈로 로드 밸런싱·캐싱·SSL 기능을 제공합니다.
- HAProxy: 고성능 로드 밸런서로 유명하며, TCP·HTTP 레벨에서 세밀한 트래픽 제어가 가능합니다.
- Traefik: 동적 환경에 최적화된 서비스 디스커버리 기반 리버스 프록시로, 컨테이너 오케스트레이션과 자연스럽게 통합됩니다.

## 결론

리버스 프록시는 현대 웹 아키텍처에서 보안·성능·운영 효율을 동시에 달성할 수 있는 핵심 컴포넌트입니다.  
도입 전 요구 사항과 트래픽 패턴을 명확히 분석하고, 고가용성·모니터링 정책을 충분히 마련한 후 구성하는 것이 중요합니다.
