## Supabase + Next.js 서버 액션

- 장점: Supabase는 Firebase 대체제로, 호스팅되는 PostgreSQL 데이터베이스와 인증, 실시간 기능 등을 일체형으로 제공하므로 초기 설정과 통합이 매우 간편. 특히 Next.js 14부터 서버 액션(Server Actions)이 안정화되어 Supabase 인증/데이터 호출을 클라이언트 코드와 같은 레벨에서 처리할 수 있게 되었고, 이 덕분에 별도의 백엔드 API를 작성하지 않아도 빠른 개발이 가능하다. 예를 들어 AI 이미지 스타트업 Pebblely는 Supabase Auth와 DB를 활용해 프로토타입을 2일 만에 구축했고, 서비스 시작 7개월 만에 100만 MAU를 돌파하는 성과를 냈다. 오픈소스 리포지토리는 GitHub에서 85.2K 스타를 기록할 정도로 활발하며, SOC2 Type2·HIPAA 인증을 획득하는 등 보안 및 규정 준수를 강조한다.
- 단점: Supabase는 기본적으로 단일 마스터 PostgreSQL 인스턴스를 사용하기 때문에 쓰기(write) 작업이 폭증하면 병목이 발생할 수 있다. 읽기 작업은 읽기 복제본을 추가해 확장할 수 있지만 쓰기는 결국 하나의 서버 성능에 의존한다. 또한 Supabase 고유의 기능(예: DB 트리거나 정책)에 비즈니스 로직을 과도하게 담으면 다른 시스템으로의 이식이 어려워지는 잠금(lock-in) 위험이 커진다. 관리자 대시보드(UI)는 아직 일부 버그가 있고 인증 흐름도 미완인 부분이 발견된다. 또한 현재 무료/초기 요금제가 주를 이루어 향후 사용자나 데이터가 늘어날 때 요금이 급격히 상승할 우려가 있다.
- 비유 및 설명: Supabase+Next.js 조합은 이미 풀옵션으로 갖춰진 렌털 아파트에 이사 가는 것과 같다. 즉, 가구(인증·DB·API)가 완비된 집에서 짐만 옮겨 놓고 바로 살 수 있으므로 개발 시작이 매우 빠르다. 그러나 집 구조를 크게 바꾸거나 특수한 설비를 설치해야 할 땐 제약이 있다. 마치 사전 제작된 방을 확장하는 형태여서, 처음엔 편하지만 ‘뼈대 자체를 뜯어고치기’는 어렵다.
- 실제 사례: Supabase는 스타트업과 빠른 프로토타이핑에 많이 활용된다. 앞서 언급한 Pebblely 사례 외에도, 인도네시아 핀테크 Xendit는 Supabase를 이용해 일주일 만에 서비스를 구축해 배포했고, 머신러닝 기반 서비스 Humata는 Supabase로 비용을 크게 절감하고 성능을 향상시켰다. 이처럼 저비용·고속 개발이 필요한 AI/스타트업 분야에서 주로 쓰인다.
- 중장기 안정성 및 확장성: Supabase는 꾸준히 성장 중이며, Next.js와 긴밀히 호환되도록 업데이트가 진행되고 있다. 읽기 부하는 쉽게 확장 가능하지만, 쓰기 트래픽이 수백만 명 규모로 폭증하면 자체 한계가 있을 수 있다. 공급자 종속성(vendor lock-in)과 미래 요금 변동성은 반드시 고려해야 한다. 그러나 Postgres 기반이므로 표준 SQL·ORM 등을 적절히 사용하고 벤더 로직 의존도를 낮추면, 장기적으로도 안정적인 데이터 관리가 가능하다.

## AWS MySQL + NestJS

- 장점: NestJS는 TypeScript 기반의 구조화된 Node.js 백엔드 프레임워크로, Angular 스타일의 DI(의존성 주입), 모듈화, 미들웨어 등을 제공해 코드 유지보수가 쉽고 생산성이 높다. 비동기 이벤트 루프 덕분에 I/O가 많은 웹 API 처리에 강점이 있으며, Express 위에서 동작해 개발자 진입장벽이 낮다. GitHub 스타도 62.6K로 높고, Adidas·GitLab·IBM·Mercedes-Benz 등 여러 글로벌 기업이 NestJS를 프로덕션에 사용하고 있다. Techempower 벤치마크에서는 순수 Node.js(113,707 req/s)보다 라우팅 오버헤드로 속도가 다소 낮았으나(38,800 req/s), 실제 웹 서비스 수준의 트래픽(수천–수만 RPS)에서는 크게 문제가 되지 않는다. 오히려 \*\*수평 확장(clustering)\*\*으로 쉽게 서버를 늘릴 수 있기 때문에, 단일 인스턴스 성능보다 여러 인스턴스 운영이 가능하다는 장점이 있다.
- 단점: NestJS는 Node 기반이므로 CPU 집약적 작업에는 비효율적이며, 복잡한 연산을 많이 수행해야 하면 다른 언어나 마이크로서비스로 분리해야 한다. 또한 Spring 같은 자바 생태계에 비해 비교적 신생 프로젝트여서, 참고할 수 있는 자료나 경험치가 적고 일부 라이브러리는 방금 생긴 정도다. 라우팅과 의존성 주입으로 인해 순수 Express보다 약간의 성능 저하는 있으나, 일반적인 웹앱에서는 크게 체감되지 않는 편이다. 보안은 기본적으로 Express 위에서 동작하므로 개발자가 JWT 인증, Passport 등의 미들웨어를 직접 구성해야 한다.
- 비유 및 설명: NestJS는 마치 레이아웃이 설계된 건축용 레고 키트와 같다. 기본 구조(모듈·컨트롤러·서비스)가 준비된 채로 제공되기 때문에, 개발자는 빌딩 블록처럼 필요한 기능을 끼워 맞춰 빠르게 개발할 수 있다. 반면, 집을 처음부터 맨손으로 짓는 것보다는 편하지만, 미리 정해진 규격에서 크게 벗어난 디자인을 적용하려면 레고 조립 이상의 노력이 필요하다.
- 실제 사례: NestJS는 주로 빠르게 성장하는 스타트업이나 중견 IT 기업에서 사용된다. 앞서 언급한 Adidas·GitLab·IBM 사례 외에도, 전자상거래 기업이 Node.js로 구축한 백엔드에 NestJS를 적용해 유지보수를 개선하기도 했다. (NestJS 사례 직접 인용자료는 한정적이지만, 상용기업 도입 수는 증가하는 추세다.) NestJS 공식 문서와 커뮤니티에 따르면, 다양한 스타트업과 중견기업에서 GraphQL 서버나 REST API에 활용되고 있다.
- 중장기 안정성 및 확장성: NestJS 자체는 오픈소스이며 활발히 업데이트되고 있고, Node.js LTS 지원과 TypeScript 지원을 이어가므로 중장기적으로도 안정된 편이다. 다만 Node.js 플랫폼 특성상 버전 업그레이드에 따른 마이그레이션이 필요할 수 있으며, 규모가 큰 서비스일 땐 Express 대신 쿠버네티스나 서버리스 등을 통한 확장을 고려해야 한다. 팀의 JavaScript/TypeScript 역량이 크다면 빠른 개발과 충분한 확장성을 동시에 누릴 수 있는 선택지다.

## AWS MySQL + Spring (Spring Boot)

- 장점: Spring Boot는 전통적으로 엔터프라이즈급 대규모 서비스에 많이 쓰이는 검증된 프레임워크다. Java 기반 멀티스레드로 초당 수많은 동시 요청을 처리할 수 있으며, Spring Security·Spring Data·Spring Cloud 등 풍부한 서브 프로젝트로 거의 모든 기능을 커버한다. GitHub 스타도 77.6K에 달해 활발한 커뮤니티 지원을 받는다. 실제 Netflix(2.3억 사용자, 매일 수십억 건의 요청 처리), Alibaba(11·11 쇼핑데이, 초당 수십만 주문 처리) 등 글로벌 서비스들이 Spring Boot로 백엔드를 구성하고 있다. Spring Security는 CSRF 방어, 세션 고정화 방지 등 기본 보안 기능이 탑재되어 있어, 민감한 거래나 데이터 처리가 많은 커머스에도 적합하다.
- 단점: Java/Spring은 개발 초기 설정이 복잡하고 학습 곡선이 높다. 초기 프로젝트 구축과 설정(Convention Over Configuration에 따른 XML/어노테이션 등)에 시간이 걸리고, 배포 시에도 JVM 튜닝이나 서버 설정이 필요하다. 실행 시 메모리 사용량이 높아 작은 규모에 비해 초기 인프라 비용이 높을 수 있다. 또한 엄청난 자유도만큼 불필요한 설정이 많아, 설정을 잘못하면 무거워질 수 있다.
- 비유 및 설명: Spring은 대규모 공장 설비와 같아서, 한번 도입하면 안정성과 확장성이 매우 뛰어나지만 도입 비용과 유지보수 노력이 크다. 큰 기계와 시스템이 모두 준비되어 있어 어떤 종류의 제품(서비스)이든 생산할 수 있는 반면, 공장 설비를 설치하고 운영하려면 많은 전문 인력과 예산이 필요하다. 즉, 폭발적 규모를 처리할 준비가 된 만큼 초기 부담도 큰 환경이다.
- 실제 사례: Spring Boot는 이미 많은 글로벌 기업에서 사용된다. 예를 들어 넷플릭스, 알리바바, 링크드인, 우버 등이 Spring Boot로 마이크로서비스를 운영 중이며, 넷플릭스의 경우 230개국 2억3천만 사용자에게 실시간 개인화 추천 등의 서비스를 제공한다. 이 외에도, eBay·Airbnb 같은 대형 플랫폼이 트래픽과 복잡도를 감당하기 위해 Spring 생태계를 선택했다.
- 중장기 안정성 및 확장성: Java 플랫폼과 Spring은 20년 넘게 검증된 스택으로, 오랜 지원과 풍부한 문서를 보장한다. AWS RDS(MySQL/Aurora)와도 친숙하게 통합 가능하며, 운영체제·클라우드 벤더의 제약도 적다. 규모가 커짐에 따라 마이크로서비스로 분리하기 쉬운 구조를 갖추고 있고, 필요시 JVM 튜닝·서버 클러스터링 등으로 성능을 지속 개선할 수 있다. 다만 초기 개발 부담과 서버 관리·비용 측면에서 작은 팀보다는 경험 많은 중대형 팀에서 장점을 발휘한다.

## 비교표

| 백엔드 조합        | 개발팀 규모(권장)     | MAU/트래픽 대응력                       | 인프라/운영 비용(상대)                  | 추천 상황                                                                                                              |
| ------------------ | --------------------- | --------------------------------------- | --------------------------------------- | ---------------------------------------------------------------------------------------------------------------------- |
| Supabase + Next.js | 1\~3명 (소규모 팀)    | 수십만\~백만 MAU, 빠른 초기 성장에 유리 | 낮음 (초기 무료/저비용, 확장 시 종량제) | 스타트업/프로토타입에 적합. 빠른 개발·프로토타이핑, 개발자 리소스 부족 시 최적. 규모가 커지면 벤더 의존과 비용을 검토. |
| AWS MySQL + NestJS | 3\~6명 (중규모)       | 수백만\~천만 MAU 이상 (수평 확장 용이)  | 중간 (AWS EC2/RDS 비용, Node.js 관리)   | 빠른 개발과 적당한 확장성을 동시에 원하는 경우. TS 기반 기술 스택에 익숙한 팀에 적합.                                  |
| AWS MySQL + Spring | 5\~10명 (대규모 가능) | 수천만 MAU 이상 (대기업 수준)           | 높음 (AWS EC2/RDS, JVM 운영비용)        | 대규모 엔터프라이즈 환경에 적합. 높은 안정성과 보안이 필요하고, Java 역량이 풍부한 팀에 권장.                          |

각 조합의 선택은 팀의 기술 역량, 개발 속도 요구사항, 예상 사용자 규모, 예산 등에 따라 달라진다. Supabase+Next.js는 소규모 팀이 빠르게 출시하고 성장 초기 비용을 절감하려 할 때 유리하다. NestJS는 JavaScript/TypeScript에 능숙한 팀이 중\~대형 서비스를 비교적 빠르게 구축하면서 확장성도 어느 정도 확보하고자 할 때 적합하다. Spring은 대규모 트래픽과 장기간 운영의 안정성을 최우선으로 하는 대기업급 서비스에 추천된다.
